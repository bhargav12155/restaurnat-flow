# Avatar and Video Generation Workflow

This document summarizes how the HeyGen integration inside this project uploads user photos, creates reusable avatars, stores them, and later generates narrated videos. It is intended for engineers who need to integrate the same flow from another app.

---

## High-Level Architecture

| Component                                               | Responsibility                                                                                     |
| ------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| Angular frontend (`src/app/…`)                          | Collects user inputs (photo, script/voice/background), calls backend endpoints.                    |
| Express proxy (`server/index.ts`)                       | Wraps HeyGen REST APIs, enforces limits, and persists metadata.                                    |
| Storage layer (`server/storage.ts`, `shared/schema.ts`) | Persists avatars, generated videos, usage counters, and notifications in Postgres via Drizzle ORM. |
| HeyGen API                                              | Accepts assets (photos/audio), produces talking-photo avatars, and renders final videos.           |

Sessions are tracked with `express-session`; anonymous visitors receive a long-lived cookie so that avatars/videos created before login can later be linked to their account.

---

## 1. Uploading a Photo & Creating an Avatar

Endpoint: `POST /api/heygen/assets` (ref. `server/index.ts`, lines ~810-990)

1. **Frontend** sends a multipart form upload with fields:
   - `file`: image binary.
   - `kind`: `'image'`.
2. **Backend** streams the file to HeyGen’s upload service: `https://upload.heygen.com/v1/asset`.
   - Adjusts the `Content-Type` header (e.g. forces `audio/x-wav` for WAV files).
3. The HeyGen response contains a public `url` and `image_key`.
4. Before creating a new avatar the backend checks `storage.getAvatarsBySessionOrUser(sessionId, userId)` to see if this session already owns an avatar with the same image. If found, it returns the existing avatar (`reused: true`).
5. Enforces the free-tier avatar limit (currently **1 per session**). If the limit is reached it returns HTTP 403.
6. When a new avatar is allowed, the server calls `POST https://api.heygen.com/v2/photo_avatar/avatar_group/create` with the `image_key`.
7. Successful responses include a `group_id` (the `talking_photo_id`). The backend persists the record via `storage.createAvatar({
   sessionId,
   userId: undefined,  // Linked later if the user logs in
   heygenAvatarId: groupId,
   imageUrl,
   avatarName: "Photo_<timestamp>"
})`.
8. Response body returned to the frontend:
   ```json
   {
     "asset_id": "02024958e1ef4e7cbd151b5ad1d1137d",
     "url": "https://resource2.heygen.ai/image/...",
     "talking_photo_id": "02024958e1ef4e7cbd151b5ad1d1137d",
     "group_id": "02024958e1ef4e7cbd151b5ad1d1137d",
     "status": "pending",
     "processing": true,
     "reused": false
   }
   ```
   Talking-photo avatars can be used immediately even if `status` is `pending`; HeyGen finalizes them internally when a video job starts.

**Persistence** (schema in `shared/schema.ts`):

```sql
avatars
  id SERIAL PK
  user_id INT (nullable, FK → users.id)
  session_id TEXT
  heygen_avatar_id VARCHAR UNIQUE  -- The group_id from HeyGen
  image_url TEXT                   -- Original first photo URL
  avatar_name VARCHAR              -- e.g., "Photo_1762365757828"
  is_trained BOOLEAN DEFAULT false -- TRUE after training completes
  created_at TIMESTAMP DEFAULT NOW
```

**Key Points:**
- Each avatar group created in HeyGen gets **one row** in the `avatars` table.
- `heygen_avatar_id` stores the `group_id` returned by HeyGen (used for all subsequent operations).
- Additional photos added via "add-looks" are stored by HeyGen, not in our database.
- `is_trained` flag indicates whether the avatar has been through the training pipeline.
- When videos reference a photo avatar, they link via `avatar_id` foreign key.

## 2. Retrieving Avatars

Endpoint: `GET /api/avatars/library` (ref. `server/index.ts`, lines ~1186-1220)

- Uses the session cookie (and optionally `userId` if logged in).
- Calls `storage.getAvatarsBySessionOrUser(sessionId, userId)` so logged-in users see both pre-login and post-login avatars.
- Returns:
  ```json
  {
    "avatars": [
      {
        "id": 12,
        "heygenAvatarId": "02024958e1ef4e7cbd151b5ad1d1137d",
        "imageUrl": "…",
        "avatarName": "Photo_1762365757828",
        "createdAt": "2024-11-05T18:05:58.000Z"
      }
    ],
    "count": 1
  }
  ```

When the user later authenticates, `storage.linkSessionToUser(sessionId, userId)` reassigns existing avatars and video records from the anonymous session to the account.

## 3. Generating a Video from an Avatar

Endpoint: `POST /api/heygen/videos` (ref. `server/index.ts`, lines ~990-1160)

**Input payload (frontend)**

```json
{
  "script": "Short narration…", // optional if custom audio uploaded
  "voiceId": "d252ca05aabc4ed5…", // HeyGen preset voice
  "voiceAssetId": null, // custom audio asset id if uploaded
  "photoAssetId": "02024958e1ef4e…", // talking photo id (from avatar upload)
  "captionEnabled": false,
  "backgroundType": "color", // or "video"/"image"
  "backgroundColor": "#101010",
  "backgroundUrl": null,
  "videoPlayStyle": "center"
}
```

**Backend steps**

1. Enforces free tier limits:
   - Maximum **3** videos per session (`storage.getVideoCountBySessionId`).
   - Maximum duration **5 seconds** for free users (estimated from script length).
2. Builds the character definition sent to HeyGen:
   ```typescript
   const character = photoAssetId
     ? { talking_photo_id: photoAssetId }
     : { avatar_id: avatarIdFromHeyGenPreset };
   const voice = voiceAssetId
     ? { voice_id: voiceId, voice_asset_id: voiceAssetId }
     : { voice_id: voiceId };
   ```
3. Constructs `video_inputs[0]` with character, voice, background settings, and caption flag.
4. Calls `POST https://api.heygen.com/v2/video/generate`.
5. Records the pending job in `video_generations` via `storage.createVideoGeneration`, storing:
   - `sessionId`, optional `avatarId` (foreign key to `avatars` table), script text, estimated duration, and HeyGen `request_id`.
6. Returns `{ request_id, status, video_url }` (URL usually null until completed).

**Status & Completion**

- Frontend polls `GET /api/heygen/videos/:requestId` (ref. lines ~1225-1285).
- Once HeyGen returns `status === 'completed'`, backend updates the original record with the final `video_url`.
- `GET /api/videos/history` returns all completed videos for the session/user.

**Video persistence** (`video_generations` table in `shared/schema.ts`)

```
video_generations
  id SERIAL PK
  user_id (nullable)
  session_id
  avatar_id (nullable FK → avatars.id)
  request_id (HeyGen video_id)
  video_url (nullable until complete)
  script_text
  duration
  created_at
```

## 4. Supporting Operations

### 4.1 Voice Library

- Endpoint: `GET /api/heygen/voices`.
- Proxies HeyGen’s `/voices` endpoint and returns the `data.voices` array (hundreds of preset voices including metadata such as locale, gender, and feature flags).

### 4.2 Script Authoring (Optional)

- Endpoint: `POST /api/heygen/scripts/auto`.
- Uses OpenAI (`gpt-4o-mini`) to produce a short script constrained to ~5 seconds of speech, respecting tone and prompt.

### 4.3 Avatar Enhancements

- `POST /api/heygen/avatars/:avatarId/motion` → adds motion to a talking photo.
- `POST /api/heygen/avatars/:avatarId/add-sound-effect` → adds background audio.
- `GET /api/heygen/avatars/:avatarId/status` → polls motion/sound-effect progress.

### 4.4 Generate New Looks (Image Variants)

- `POST /api/heygen/avatars/:groupId/generate-look`
  - Sends `prompt`, `orientation`, `pose`, `style` to HeyGen (`/photo_avatar/look/generate`).
  - Records usage in `generate_look_usage` (see storage logic) and guides clients to poll for `generation_id` status.

### 4.5 Avatar Groups & Multi-Photo Training

HeyGen supports **avatar groups** where you upload multiple photos of the same person to create a higher-quality, trained avatar. This significantly improves lip-sync accuracy, facial expressions, and overall realism.

#### Adding Photos to an Existing Avatar Group

**Endpoint:** `POST /api/heygen/avatars/:groupId/add-looks`

This endpoint accepts multiple image files and adds them to an existing avatar group created during the initial photo upload.

**Process:**
1. Frontend sends multipart form with up to 10 files (recommended 2-4).
2. Backend uploads each file to HeyGen's asset service and extracts `image_key` from URLs.
3. Calls `POST https://api.heygen.com/v2/photo_avatar/avatar_group/add` with:
   ```json
   {
     "group_id": "02024958e1ef4e7cbd151b5ad1d1137d",
     "image_keys": ["image/abc123.jpg", "image/def456.jpg"],
     "name": "Enhanced_1762370000000"
   }
   ```
4. Returns success confirmation with number of looks added.

**Important Constraints:**
- HeyGen limits uploads to **4 photos at a time** (per API call).
- Cannot add photos while training is in progress (10-30 min wait).
- Photos must be of the **same person** for best training results.
- Recommended: high-quality images, different angles/expressions, good lighting.

**Response:**
```json
{
  "success": true,
  "group_id": "02024958e1ef4e7cbd151b5ad1d1137d",
  "added_looks": 3,
  "image_keys": ["image/abc.jpg", "image/def.jpg", "image/ghi.jpg"],
  "message": "Added 3 looks to avatar group. Consider training for best results."
}
```

#### Training an Avatar Group

**Endpoint:** `POST /api/heygen/avatars/:groupId/train`

Training processes all photos in the group to create a unified, high-fidelity avatar model.

**Process:**
1. Frontend sends training request with `group_id`.
2. Backend calls `POST https://api.heygen.com/v2/photo_avatar/train`:
   ```json
   {
     "group_id": "02024958e1ef4e7cbd151b5ad1d1137d"
   }
   ```
3. HeyGen returns a `job_id` and begins async processing.
4. Training takes **10-30 minutes** depending on server load.

**Requirements:**
- Minimum **2 high-quality photos** of the same person.
- Cannot start training if another training job is already running.
- Trained avatars can still be used for video generation during training (uses best available version).

**Response:**
```json
{
  "success": true,
  "group_id": "02024958e1ef4e7cbd151b5ad1d1137d",
  "job_id": "train_abc123",
  "status": "training",
  "message": "Avatar training started. Use GET /api/heygen/avatars/:groupId/training-status to check progress."
}
```

#### Checking Training Status

**Endpoint:** `GET /api/heygen/avatars/:groupId/training-status`

Polls the current training state of an avatar group.

**Process:**
1. Backend calls `GET https://api.heygen.com/v2/photo_avatar/avatar_group/:groupId`.
2. Extracts status from response.

**Response:**
```json
{
  "group_id": "02024958e1ef4e7cbd151b5ad1d1137d",
  "status": "completed",
  "trained": true,
  "training": false,
  "details": {
    "status": "completed",
    "photo_count": 4,
    "created_at": "2024-11-05T10:30:00Z",
    "completed_at": "2024-11-05T10:45:00Z"
  }
}
```

**Possible status values:**
- `pending`: Training queued but not started.
- `training`: Currently processing (10-30 min).
- `completed`: Training finished successfully.
- `failed`: Training encountered an error.

#### Database Tracking

The `avatars` table includes an `is_trained` boolean column to track whether an avatar group has completed training:

```sql
UPDATE avatars 
SET is_trained = true 
WHERE heygen_avatar_id = '02024958e1ef4e7cbd151b5ad1d1137d';
```

**Workflow Integration:**
1. User uploads initial photo → `avatars` record created with `is_trained = false`.
2. User adds more photos → No database update (group metadata lives in HeyGen).
3. User triggers training → Backend can mark `is_trained = 'pending'` (optional).
4. Training completes → Frontend checks status, backend updates `is_trained = true`.
5. Videos generated with trained avatars have visibly higher quality.

**Best Practices:**
- Show training status in UI with progress indicator.
- Disable "Add Photos" and "Train" buttons while training is active.
- Send email notifications when training completes (see `server/email-service.ts`).
- Allow video generation during training (HeyGen handles gracefully).

## 5. Data Flow Summary

1. **Photo Upload** → `POST /api/heygen/assets` → HeyGen asset service → `avatars` table (single photo).
2. **Avatar Library** → `GET /api/avatars/library` → `avatars` table (session/user filtered).
3. **Add More Photos** → `POST /api/heygen/avatars/:groupId/add-looks` → uploads multiple images → adds to HeyGen group.
4. **Train Avatar** → `POST /api/heygen/avatars/:groupId/train` → starts 10-30 min training job.
5. **Check Training** → `GET /api/heygen/avatars/:groupId/training-status` → polls until `status === 'completed'`.
6. **Script (optional)** → `POST /api/heygen/scripts/auto` → OpenAI response.
7. **Voice Selection** → `GET /api/heygen/voices` → HeyGen voice catalog.
8. **Video Generation** → `POST /api/heygen/videos` → HeyGen video job → `video_generations` table.
9. **Status Polling** → `GET /api/heygen/videos/:requestId` → updates video URL once finished.
10. **History Listing** → `GET /api/videos/history` → completed jobs for UI playback/download.

All items created pre-login remain associated with the anonymous `sessionId`. Upon login or registration, `storage.linkSessionToUser` migrates them to the user record so that workflows remain seamless.

## 6. Reusing the Flow in Another Application

To integrate these steps elsewhere:

1. **Reuse proxy endpoints** or port the logic from `server/index.ts`:
   - `POST /api/heygen/assets` → upload single photo, create group.
   - `POST /api/heygen/avatars/:groupId/add-looks` → add 2-4 more photos.
   - `POST /api/heygen/avatars/:groupId/train` → start training job.
   - `GET /api/heygen/avatars/:groupId/training-status` → poll completion.
   - `POST /api/heygen/videos` → generate video with trained avatar.

2. **Mirror database schema** for `avatars` and `video_generations` (see `shared/schema.ts`):
   - Track each avatar group with `heygen_avatar_id` (HeyGen's `group_id`).
   - Store `is_trained` boolean to show trained vs. untrained avatars differently in UI.
   - Link videos to avatars via foreign key for usage analytics.

3. **Maintain session identifier** (cookie or token) to support pre-login usage:
   - Anonymous users can create avatars and videos.
   - Upon login, migrate session data to user account via `storage.linkSessionToUser`.

4. **Enforce limits** as needed (free-tier logic is centralized in the proxy):
   - 1 avatar per free session.
   - 3 videos per free session.
   - 5-second max duration for free users.

5. **Always store HeyGen identifiers:**
   - `talking_photo_id` / `group_id` for avatars.
   - `video_id` (`request_id`) for videos.

6. **Poll HeyGen for completion:**
   - Training: poll `training-status` endpoint every 10-30 seconds until `status === 'completed'`.
   - Videos: poll video status endpoint until `status === 'completed'` and `video_url` is available.
   - Update database records once complete.

7. **Handle multi-photo workflow:**
   - Single photo → instant avatar creation (untrained).
   - Add 2-4 more photos → still usable immediately.
   - Train → 10-30 min wait → significantly better quality.
   - Show training progress in UI to set expectations.

8. **Email notifications** (optional but recommended):
   - Send completion email when training finishes (see `server/email-service.ts`).
   - Include `group_id` and link to video creation page.

This documentation should give another team enough context to reproduce the end-to-end avatar/video workflow, including the advanced multi-photo training pipeline, without diving into the entire codebase. Reach out if you need sequence diagrams or SDK wrappers.
